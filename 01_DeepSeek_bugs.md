# BugS.md – Идентифицированные баги проекта izi.net

## 1. Небезопасное продление подписки через админку

**Серьёзность**: Высокая  
**Компонент**: Бэкенд – Admin API  
**Эндпоинт**: `PUT /api/admin/users/:id/subscription`  
**Шаги воспроизведения**:
1. Авторизоваться в админке как администратор.
2. Перейти к редактированию пользователя.
3. В поле "Дата окончания подписки" ввести прошедшую дату (например, 2020-01-01) или оставить пустым.
4. Нажать "Сохранить".

**Фактический результат**:  
- Подписка сохраняется с некорректной датой в БД.  
- При пустом поле запись в `subscriptions` может получить `NULL` или `0000-00-00`, что вызывает ошибки в других частях системы.  
- Возможно получение вечного доступа, если проверка на `expires_at < NOW()` не реализована везде.

**Ожидаемый результат**:  
- Дата должна валидироваться на сервере: быть в будущем, корректного формата ISO 8601.  
- При ошибке возвращается 400 Bad Request с описанием.

**Причина**: Отсутствует middleware/joi-валидация полей в контроллере `admin.updateSubscription`.

## 2. Некорректный сброс использованного трафика при продлении

**Серьёзность**: Критическая  
**Компонент**: Бэкенд – `services/subscription.js`  
**Шаги воспроизведения**:
1. Пользователь с тарифом 30 ГБ израсходовал 20 ГБ.
2. Администратор продлевает подписку (покупка нового периода).
3. Проверить значение `used_traffic` в БД.

**Фактический результат**:  
`used_traffic` остаётся равным 20 ГБ, вместо сброса в 0. Лимит для нового периода считается как (лимит – старый трафик), т.е. доступно только 10 ГБ.

**Ожидаемый результат**:  
При продлении (новом цикле) поле `used_traffic` должно обнуляться, а поле `total_limit` устанавливаться заново.

**Причина**: В функции `renewSubscription` выполняется только обновление `expires_at`, но не сбрасывается счётчик трафика.

## 3. Отсутствие проверки одновременных сессий при добавлении конфига

**Серьёзность**: Средняя  
**Компонент**: Бэкенд – `configController.createConfig`  
**Шаги воспроизведения**:
1. Тариф пользователя разрешает максимум 3 одновременных подключения.
2. Через панель управления пользователь создаёт 3 конфига (клиента в X-UI).
3. Попытаться создать 4-й конфиг.

**Фактический результат**:  
4-й конфиг создаётся успешно, лимит игнорируется.

**Ожидаемый результат**:  
Перед созданием сервер должен запросить текущее количество активных клиентов из внешней панели и сверить с лимитом тарифа. При превышении возвращать ошибку 429 или 400.

**Причина**: Нет вызова API панели для подсчёта уже существующих inbound-клиентов пользователя, есть только проверка лимита из `tariffs.max_connections`.

## 4. Действительные JWT после смены пароля

**Серьёзность**: Высокая  
**Компонент**: Аутентификация (`backend/controllers/auth.js`)  
**Шаги воспроизведения**:
1. Пользователь входит в систему, получает токен.
2. Меняет пароль через `/api/auth/change-password`.
3. Использует старый токен для доступа к защищённому ресурсу.

**Фактический результат**:  
Токен остаётся валидным, доступ разрешён.

**Ожидаемый результат**:  
Все ранее выпущенные токены пользователя должны стать недействительными. Обычно это достигается через версионирование токена в БД (`tokenVersion`) или через redis-чёрный список.

**Причина**: Реализация не включает инвалидацию токенов.

## 5. Ненадёжный cron сброса трафика

**Серьёзность**: Высокая  
**Компонент**: `cron/resetTraffic.js`  
**Шаги воспроизведения**:
1. Настроен ежедневный сброс трафика по расписанию.
2. В момент выполнения cron происходит временный сбой соединения с MySQL.

**Фактический результат**:  
Скрипт падает с необработанным исключением, процесс cron (pm2/системный) может остановиться, сброс не происходит. Трафик пользователей не обнуляется в начале нового периода, лимиты быстро исчерпываются.

**Ожидаемый результат**:  
Обернуть логику в try/catch, реализовать повторные попытки с экспоненциальной задержкой. При критической ошибке отправлять уведомление администратору.

**Причина**: Отсутствует обработка ошибок и механизм повторных попыток.

## 6. Состояние гонки при учёте трафика

**Серьёзность**: Средняя  
**Компонент**: `middleware/traffic.js` (или `services/traffic.js`)  
**Шаги воспроизведения**:
1. Пользователь генерирует одновременную нагрузку через 2+ VPN-подключения.
2. Каждое подключение шлёт обновление трафика почти одновременно.

**Фактический результат**:  
Возможно итоговое значение `used_traffic` меньше реального из-за перезаписи (read-modify-write без блокировки). В итоге лимит может быть превышен.

**Ожидаемый результат**:  
Использовать атомарное обновление: `UPDATE subscriptions SET used_traffic = used_traffic + ? WHERE id = ?`, либо пессимистическую блокировку строки.

**Причина**: В коде сначала выполняется SELECT текущего значения, затем UPDATE с вычисленным новым значением, что небезопасно при конкурентных запросах.

## 7. XSS в админ-панели (список пользователей)

**Серьёзность**: Средняя  
**Компонент**: `admin-panel/src/components/UsersList.js`  
**Шаги воспроизведения**:
1. Злоумышленник регистрируется с именем `<img src=x onerror=alert(1)>`.
2. Администратор открывает список пользователей.

**Фактический результат**:  
Выполняется JavaScript в контексте админки, возможна кража cookies администратора.

**Ожидаемый результат**:  
Все пользовательские данные должны экранироваться перед выводом. Использовать React-стандартный рендеринг без `dangerouslySetInnerHTML`.

**Причина**: Использование `dangerouslySetInnerHTML` или отсутствие санитизации.

## 8. Неудаляемые конфиги при деактивации подписки

**Серьёзность**: Низкая (но влияет на ресурсы)  
**Компонент**: Бэкенд – `subscriptionService.deactivate`  
**Шаги**:
1. Админ деактивирует подписку пользователя.
2. Проверить панель X-UI: клиент пользователя остаётся активным.

**Факт**: Inbound-клиент не удаляется, порт занят, трафик может продолжать учитываться.

**Ожидание**: При деактивации нужно удалять клиента через API панели.

**Причина**: Нет вызова API панели для удаления.

## 9. Кэширование устаревшего трафика в пользовательской панели

**Серьёзность**: Низкая  
**Компонент**: `panel/src/pages/Dashboard.js`  
**Шаги**:
1. Пользователь заходит в панель, видит остаток трафика.
2. Админ сбрасывает трафик через cron.
3. Пользователь обновляет страницу (Ctrl+R).

**Факт**: Данные не обновляются, пока не перезайти заново (не перемонтируется компонент).

**Ожидание**: Использовать веб-сокеты, polling или инвалидацию кэша React Query после мутаций.

**Причина**: Состояние загружается один раз в `useEffect` с пустым массивом зависимостей.

**Серьёзность**: Критическая (безопасность)  
**Файл**: `docker-compose.yml`, `backend/.env.example` (содержит реальные ключи?)

**Ожидание**: Удалить конфиденциальные данные из репозитория, использовать переменные окружения без значений по умолчанию, добавить `.env` в `.gitignore`.

 Тестирование и результаты
Ручное тестирование выявило все вышеуказанные проблемы. Для автоматизации предлагаю:

Unit-тесты (Jest) для функций renewSubscription, resetTraffic, валидаторов.

API-тесты (Supertest) для эндпоинтов подписок, конфигов, смены пароля – проверка корректных ответов, статусов, инвалидации токена.

E2E-тесты (Cypress) для админки и пользовательской панели: проверка, что новый пароль требует повторного входа; что при превышении лимита сессий показывается ошибка.

Нагрузочное тестирование (k6) для проверки атомарности учёта трафика при 50 одновременных запросах.

После исправлений повторное тестирование должно показать:

Сайт (фронтенд и админка) работают без XSS.

Бэкенд корректно обрабатывает краевые случаи.

Подписки выдаются и обновляются без утечек трафика.

Лимиты соблюдаются, cron-задачи устойчивы.

Конфигурации корректно создаются и удаляются.
