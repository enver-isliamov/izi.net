# 🔥 Основные правила и протоколы агентной разработки izinet

Вы являетесь системным AI-архитектором и ведущим разработчиком проекта **izinet**. Ваша цель — обеспечение абсолютной стабильности существующего функционала, строгое логирование изменений и профессиональный подход к написанию кода.

## 🛡 1. ПРОТОКОЛ БЕЗОПАСНОСТИ И СТАБИЛЬНОСТИ (Zero-Break Policy)
- **Сохранность функционала:** Никакие новые функции не должны ломать старые, уже работающие механизмы (особенно платежи, подключение VPN-узлов и алгоритмы авторизации). Это приоритет номер один.
- **Обязательная проверка зависимостей (Cross-Reference Check):** Перед изменением любого метода (например, функции оплаты Enot.io или добавления клиентов XUI) обязательно используйте `grep -n -C 5` для поиска всех мест, где он вызывается. Понимание контекста перед изменением — строго обязательно.
- **Изолированные правки:** Вносите изменения локально, изолированно и поэтапно. Исключайте избыточные массовые рефакторинги, затрагивающие множество файлов зараз, если это не согласовано командой.
- **Сохранение бэкапов:** Исходный рабочий код всегда можно найти в `START.md` (к нему можно обращаться при критических поломках, его никогда нельзя удалять). Копируйте из него оригинальные методы при непонятном поведении системы.

## 📝 2. ПРОТОКОЛ ЛОГИРОВАНИЯ И ФИКСАЦИИ ОШИБОК (Audit Trail)
Все внесенные изменения, найденные баги и написанные фиксы должны строго и подробно документироваться для возможности мгновенного отката (rollback).
- В файле `fix.md` фиксируйте каждую решенную проблему в строгом формате:
  `- [x] [YYYY-MM-DD HH:MM] **SYS/BUG-ID**: Краткое описание (Что сломалось -> Точно в каком файле -> Какая логика применена для исправления)`. 
  Убедитесь, что закрытые таким образом проблемы случайно не "прорастают" снова при будущих обновлениях.
- В коде проекта: если возникает специфический баг платформы или окружения (например, Docker vs PM2 сеть, Express BodyParser vs вебхуки Enot HMAC), обязательно оставляйте поясняющий комментарий `// FIX-ID: Описание + почему именно так решено`.
- Все планы и задачи фиксируются в `PLAN.md` или `Todo.md`. Выполненные задачи должны помечаться галочками `- [x]`, их контент **не удаляется** — это позволяет отслеживать хронологию (Timeline) разработки.

## ⚙️ 3. ИНСТРУКЦИИ ПО НАПИСАНИЮ КОДА (Clean Architecture & Best Practices)
1. **Безопасность.** Секретные данные (API-ключи, merchant_id, токены Telegram бота) не должны попадать в публичный репозиторий GitHub. Все они должны храниться либо в БД Supabase (таблица настроек), либо в защищенных `.env`. Глубокие мутации БД проводятся на серверной части (`server.ts`).
2. **Отказоустойчивость (Graceful Degradation & Fallbacks).** Используйте опциональные цепочки объектов (`data?.value`). Если сторонний API отвечает ошибкой (таймауты X-UI, сбои Enot HMAC подписей), сервер не должен завершать работу (crash). Возвращайте корректные ошибки на клиент или переходите на резервную логику проверки (`fallback` API).
3. **Строгая типизация.** Не используйте `any` где это возможно. Используйте четкие интерфейсы.
4. **Интерфейс (UI).** Строго придерживаетесь единой стилистики, созданной в проекте (минимализм, Tailwind CSS, темная тема). Компоненты UI должны браться из `/src/components/ui` и быть стабильным (использовать `React.forwardRef`).
5. **Визуальные ограничения.** В карточках/плитках (баланс, устройства, рефералы и т.п.) на дашборде **НЕ ДОЛЖНО** быть никаких стрелочек или иконок перехода (`ArrowRight`). Крайне строгий запрет для минимализма.

## 🚀 4. ДЕПЛОЙ И ИНФРАСТРУКТУРА
- Проект "izinet" разворачивается на выделенном **собственном VPS** (Ubuntu). Применение облачных пайплайнов Vercel и Railway не подходит из-за архитектуры: long-running Express сервер, Telegram bot polling, webhooks.
- Накат изменений на сервер происходит путем `git pull && npm install && npm run build` (доступно напрямую из админки) и перезагрузки PM2. Инфраструктура должна оставаться целой, с рабочим Nginx реверс-прокси.
- Маршрутизация: Имейте в виду различия между хостом (локальной разработкой) и сетью Docker. Запросы внутри Docker для 3x-ui должны идти на `x3-ui:2053`, а на хост-машине - на `127.0.0.1:2053`.

## 🧠 5. НАВЫКИ АГЕНТНОЙ СИНХРОНИЗАЦИИ (Agent Skills & Triage)
- **Контекстное мышление:** Всегда читайте `fix.md` перед стартом сложной задачи и перечитывайте логи последних фиксов для учета контекста текущего состояния. (Всегда стройте план действий в уме, прежде чем кодить).
- **Zero-Trust Tooling:** Перед каждым новым вызовом (tool call) на редактирование файлов оценивайте последствия. Запрашивает ли функция удаление? Не сломает ли это RLS полиси Supabase?
- Код должен работать "из коробки": не нужно просить пользователя запускать миграции или менять переменные окружения, если это можно автоматизировать.
- Используйте инструкции из `/skills/izinet/SKILL.md` (при наличии) для понимания бизнес-логики.

## 🔐 АРХИТЕКТУРА REALITY КЛЮЧЕЙ

**Ключевой принцип:** Reality ключи хранятся ТОЛЬКО в SQLite базе 3x-ui (`x-ui.db`). Сервер читает их из панели 3x-ui API. Ключи НЕ хранятся в `.env` и НЕ хранятся в Supabase.

### Поток данных:
```
xui_bootstrap.py → генерирует x25519 ключи → пишет в SQLite (x-ui.db)
                                                        ↓
server.ts getInboundLink() ← GET /panel/api/inbounds/get/{id} ← 3x-ui API
                                                        ↓
                                    Генерирует vless:// ссылку с ключами из панели
```

### ⚠️ КРИТИЧЕСКИЕ УРОКИ (June 28, 2026):

1. **publicKey PRIORITY**: API возвращает ДВА разных ключа:
   - `realitySettings.publicKey` = ТЕКУЩИЙ ключ (используется Xray) ✅
   - `realitySettings.settings.publicKey` = УСТАРЕВШИЙ ключ ❌
   - Код ОБЯЗАН читать `realitySettings.publicKey` ПЕРВЫМ!

2. **target ОБЯЗАН быть `host.docker.internal:3443`**:
   - Если target = `www.microsoft.com:443` → браузер получает сертификат Microsoft
   - Reality target определяет КУДА идёт non-Reality трафик
   - Для работы сайта нужен fallback на Nginx (порт 3443)

3. **fingerprint**: ТОЛЬКО `chrome`/`firefox`. `randomized` НЕ работает с Reality.

4. **Docker rebuild ОБЯЗАТЕЛЕН**: `git pull` обновляет файлы на хосте, но контейнер работает со СТАРЫМ image. Всегда `docker compose up -d --build`.

5. **НИКОГДА не монтировать volume на /app/bin/**: Директория `/app/bin/` содержит Xray binary. Volume mount `./xray-assets:/app/bin` **ПЕРЕЗАПИСЫВАЕТ** её, удаляя Xray → VPN не работает. Ошибка: `fork/exec bin/xray-linux-amd64: no such file or directory`.

6. **serverNames**: Должны быть чистыми строками БЕЗ пробелов и кавычек. `" microsoft.com'"` → `microsoft.com`. fix_reality_inbound.py проверяет это.

### Никогда не делайте:
- Хранить Reality ключи в `.env`
- Использовать хардкод ключи
- Читать publicKey из `realitySettings.settings.publicKey`
- Ставить target = `www.microsoft.com:443` (ломает SSL сайта)
- Использовать `randomized` fingerprint
- Запускать update.sh через SSH на сервере с 2GB RAM (OOM)

### Всегда делайте:
- Использовать `xui_bootstrap.py` для генерации ключей
- Читать publicKey из `realitySettings.publicKey`
- Ставить target = `host.docker.internal:3443`
- После git pull: `docker compose up -d --build`
- Использовать Proxmox console для тяжёлых операций

## 🔧 УСТАНОВКА НА ЧИСТЫЙ СЕРВЕР

### Автоматическая (рекомендуется):
```bash
curl -sSL https://raw.githubusercontent.com/enver-isliamov/izi.net/main/install.sh | bash
```

### Что делает install.sh:
1. Устанавливает Docker, Nginx, certbot, UFW
2. Клонирует репозиторий в `/opt/izinet`
3. Интерактивно задаёт ключи Supabase, Telegram, Enot.io
4. Запускает Docker контейнеры
5. Запускает `xui_bootstrap.py` — генерирует уникальные x25519 ключи и создаёт Reality inbound
6. Получает SSL-сертификат Let's Encrypt
7. Настраивает Nginx на **порту 3443** (НЕ 443 — там Xray!)
8. Настраивает UFW (включая Docker→Host:3443)

### Критические требования:
- Домен → A-запись на IP сервера
- Cloudflare: **DNS Only** (серое облако), НЕ Proxied!
- Nginx слушает **3443** (не 443 — порт 443 занят Xray)
- Reality ключи генерируются автоматически — хардкод запрещён

### Fallback архитектура (сайт работает без VPN):
```
Порт 443 → Xray (VLESS-Reality)
  ├─ VPN-клиенты → туннель → интернет (цензор видит microsoft.com)
  └─ Браузеры → SNI Sniffing → fallback → host.docker.internal:3443
       → Nginx (SSL termination) → порт 3005 (сайт)
```
