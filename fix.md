## 🔴 ПРОБЛЕМЫ МУЛЬТИСЕРВЕРНОЙ АРХИТЕКТУРЫ (UNIVERSAL SUBSCRIPTION)

В ходе анализа выявления проблем с "бесшовным переключением между серверами", обнаружены фундаментальные уязвимости в текущей имплементации:

### BUG-08: Ре-синхронизация юзеров (Legacy users)
Старые пользователи, или те, кто купил подписку до внедрения мульти-серверности, имеют в БД `v2ray_config` только с одним ключом (от одного сервера).
**Проблема:** Нет API или фоновой задачи (Cron), которая бы просканировала всех пользователей на сервере, сгенерировала бы им ключи на недостающих активных серверах и обновила бы их `v2ray_config` в БД.
**Решение:** (РЕШЕНО) Добавлена кнопка "Синхронизировать юзеров" в `/admin/servers`, вызывающая `/api/admin/system/sync-servers` для прохода по всем активным подпискам и восстановления недостающих линков.

### BUG-09: Добавление новых VPN серверов администратором
Если администратор добавит новый VPN сервер из панели администратора, он становится доступен только для *новых* покупок или переподписанных устройств.
**Проблема:** Существующие пользователи не появятся на новом сервере автоматически, пока не произведут продление или покупку, поэтому их "универсальная ссылка" не получит новый узел.
**Решение:** (РЕШЕНО) После добавления сервера администратор теперь может нажать "Синхронизировать юзеров", чтобы добавить новые ключи на новый сервер для всех существующих пользователей.

### BUG-10: Отсутствие механизма Retry при сбоях (Timeout / Network Failures)
В функции `app.post('/api/subscription/buy')` клиенты `v2ray` создаются параллельно на всех серверах через цикл. Если один из серверов не отвечает (таймаут), возвращается `error`, но процесс продолжает сохранять успешные ключи.
**Проблема:** У юзера сохраняется подписка в БД без одной из локаций. Он не знает о проблеме и никогда туда не попадет.
**Решение:** (РЕШЕНО) На дашборде пользователя при первичной загрузке в фоне вызывается автоматическая самопроверка `/api/subscription/sync-servers`, восстанавливающая ключи, если количество конфигураций `v2ray_config` не совпадает с пулом серверов.

### BUG-11: Мертвые ключи при отключении/удалении серверов
Если сервер выходит из строя и админ помечает его `is_active: false`, старые конфигурации все еще отдаются в `/api/sub/:id`.
**Проблема:** Юзер продолжает получать мертвые URL-адреса, а клиент v2ray может тратить время на попытки подключения, что приводит к обрывам или замедлениям у пользователей.
**Решение:** (РЕШЕНО) `/api/sub/:id` теперь подгружает список активных серверов, парсит `#<remark>` (добавленный к ключам) и возвращает только конфиги от серверов `is_active = true`.

## BUG-01: VPS backend не синхронизирован с security-правками

Статус: [x] ИСПРАВЛЕНО. 
Все API-эндпоинты теперь проверяют `Authorization` заголовок ДО валидации параметров тела запроса.

Как протестировать:
1. Выполните запрос без заголовка Authorization: 
   `curl -X POST http://194.50.94.28:3005/api/subscription/buy -d '{"planId": "test"}'`
2. Ожидаемый результат: `401 Unauthorized`.
3. (Раньше возвращалось `400 Missing parameters`).

## BUG-02: Пополнение баланса Enot.io (Процесс вебхука)

Статус: [ВЫПОЛНЕНО]. 
1. Внедрена гибкая проверка подписи.
2. Добавлен маппинг `userId` / `user_id`.
3. Создан инструмент в Админке (Платежи) для ручного зачисления застрявших оплат.
4. Добавлено логирование `[EnotDebug]` для отладки подписи в будущем.

## BUG-03: Удаление дополнительного устройства нужно проверить после деплоя

Статус: локально исправлено, требуется deploy + проверка.

Проблема:
- клиент в 3x-ui должен удаляться при удалении устройства из UI.

Что проверить после деплоя:
1. Добавить дополнительное устройство.
2. Удалить его из UI.
3. Убедиться, что в 3x-ui клиент удален.
