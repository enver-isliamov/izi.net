# Архитектура бесшовного перенаправления и сожительства Reality (443) + Nginx (3443)

Этот документ описывает техническое решение, которое позволяет одновременно запускать на одном публичном порту **443** два ключевых сервиса:
1. **VPN-протокол VLESS-Reality** (маскирующийся под внешние доверенные сайты).
2. **Веб-сайт личного кабинета и REST API** (защищенные SSL-сертификатом Let's Encrypt).

Данное решение полностью автоматизировано и работает "из коробки" при развертывании.

---

## 1. Суть технического решения (Co-existence)

Обычно запуск VPN на порту `443` мешает работе веб-сервера (Nginx/Apache), так как порт может быть занят только одним процессом. Мы решили эту проблему по следующей схеме:

```
                          [ Трафик на порт 443 ]
                                    │
                                    ▼
                       [ Xray-Core (VLESS-Reality) ]
                       Слушает 443 порт на хосте
                                    │
            ┌───────────────────────┴───────────────────────┐
            ▼                                               ▼
   [ Reality VPN Трафик ]                        [ Обычный HTTPS Трафик ]
   (Авторизованные клиенты,                      (Браузеры, API-запросы,
    использующие UUID)                           поисковые боты к изинет)
            │                                               │
            ▼                                               ▼
       Проксируется и                                  Перенаправляется через
    успешно шифруется внутри                        **Reality Target/Dest** +
        Xray тоннеля                                   **VLESS Fallbacks**
                                                            │
                                                            ▼
                                               [ host.docker.internal:3443 ]
                                               (Локальный Nginx на хосте со
                                                SSL Let's Encrypt сертификатом)
                                                            │
                                                            ▼
                                               [ 127.0.0.1:3005 (Express) ]
                                               (Наш сайт и API-бекенд)
```

### Почему раньше возникала ошибка Google 404?
В стандартных настройках Reality в качестве маскировочного узла (`target` и `dest`) указывается сторонний домен (например, `www.google.com` или `dl.google.com`). 
Если на порт `443` приходил обычный GET-запрос от браузера, пытающегося открыть `https://izinet.online`, Xray-Core послушно пересылал этот запрос на Google. Google видел заголовок `Host: izinet.online`, понимал, что этот домен ему не принадлежит, и возвращал стандартную ошибку **Google 404**.

## 2. Автоматическое исправление в `repair_xui.py`

Для полной автоматизации и устранения ручных ошибок разработан скрипт `repair_xui.py`, который напрямую модифицирует базу данных панели 3x-ui (`x-ui.db`) и перезапускает сервисы.

### Ключевые изменения в SQLite базе:

1. **Reality Target и Dest перенаправлены на Nginx:**
   Параметры `target` и `dest` в настройках `realitySettings` внутри `inbounds` изменены с внешних серверов на локальный порт Nginx:
   ```json
   "target": "host.docker.internal:3443",
   "dest": "host.docker.internal:3443"
   ```
   Это заставляет Xray отправлять любой неопознанный HTTPS-трафик на наш собственный веб-сервер.

2. **Точные правила Fallbacks:**
   В протокол fallbacks встроены 3 строгих правила, которые направляют трафик на расшифровку в Nginx:
   * **Правило 1:** Для домена `izinet.online` -> перенаправление на `host.docker.internal:3443`.
   * **Правило 2:** Для домена `www.izinet.online` -> перенаправление на `host.docker.internal:3443`.
   * **Правило 3 (Wildcard):** Дефолтное перенаправление без указания SNI -> на `host.docker.internal:3443`.

3. **Принудительное включение SNI Sniffing:**
   Поле `sniffing` в базе данных принудительно переводится в активный статус с распознаванием `tls` и `http` протоколов:
   ```json
   {
     "enabled": true,
     "destOverride": ["http", "tls"],
     "routeOnly": false
   }
   ```
   Без этого Xray-Core не сможет прочесть SNI-заголовок (имя домена) из пакета TLS ClientHello, что сделало бы fallbacks нерабочими.

## 3. Настройки DNS Cloudflare (Критически важно!)

Чтобы архитектура Reality успешно функционировала, ваш домен должен обращаться к серверу напрямую:

1. Откройте панель управления **Cloudflare**.
2. В разделе **DNS** проверьте записи типа `A` для:
   - `izinet.online`
   - `www.izinet.online`
3. Установите **Proxy Status** в режим **"DNS Only"** (серое облако, а не оранжевое!).

> ⚠️ **ПОЧЕМУ ЭТО ВАЖНО:** Если облако оранжевое (Proxied), Cloudflare терминирует SSL-соединение на своих серверах. Reality-клиенты отправляют специфические TLS-запросы со своими ключами, которые Cloudflare не понимает и сбрасывает соединение, выдавая таймаут на стороне пользователя.

## 4. Чек-лист проверки и обслуживания

Если вы переносите проект на другой сервер или разворачиваете новую ноду, выполните цепочку действий:

```bash
# 1. Сделайте пулл свежего кода и скриптов
cd /opt/izinet && git pull origin main

# 2. Выдайте права на исполнение
chmod +x diagnose.sh repair_xui.py

# 3. Запустите автоматический ремонтник конфигурации
sudo python3 repair_xui.py

# 4. Проверьте статус портов и доступность сайта
sudo bash diagnose.sh
```

### Как должен выглядеть правильный вывод диагностики:
* **Порт 443:** Занят процессом `docker-proxy` (контейнер `x3-ui` с Xray-Core).
* **Порт 3443:** Занят локальным процессом `nginx` на хосте.
* **Порт 3005:** Занят процессом `docker-proxy` (контейнер `izinet-app` с Express-бекендом).
* **Сайт `https://izinet.online`:** Открывается корректно (Nginx успешно принимает fallback-трафик, расшифровывает сертификатом Let's Encrypt и отдает Express-бэкенду).
* **Панель 3x-ui:** Доступна по адресу `http://YOUR_IP:2053/` (или настроенному кастомному порту).

Это решение гарантирует 100% стабильность работы сайта и VPN-протоколов нового поколения на одном сервере.
