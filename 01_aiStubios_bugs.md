# Реестр обнаруженных ошибок и багов (BugS.md)

В данном документе собраны критические дефекты, влияющие на работоспособность клиентской части, панели администратора, генерации конфигураций и механизмов контроля лимитов.

---

## 1. Блок: Выдача и обновление подписок (Subscription Issuance & Update)

### BUG-001: Сбой обновления конфигурации подписки на клиенте (кэширование заголовков)
* **Описание:** При обновлении клиентом подписки через API (например, запрос на `/api/v1/subscription/update/{token}`) возвращаются устаревшие данные из-за некорректной настройки кэширования на стороне бэкенда или веб-сервера (Nginx).
* **Действия для воспроизведения (Шаги):**
  1. Авторизоваться на сайте под тестовым пользователем.
  2. Получить ссылку на подписку.
  3. В админ-панели изменить параметры подписки (например, добавить новый сервер).
  4. Выполнить HTTP-запрос (GET) по ссылке подписки со стороны клиента.
  5. Повторить запрос через 1 минуту.
* **Фактический результат:** Бэкенд возвращает старый конфигурационный файл (304 Not Modified или старый кэшированный 200 OK), изменения не применяются у клиента.
* **Ожидаемый результат:** Бэкенд должен отдавать актуальную конфигурацию. Заголовки ответа должны содержать `Cache-Control: no-store, no-cache, must-revalidate` для предотвращения кэширования промежуточными прокси-серверами.

---

### BUG-002: Race Condition при выдаче подписки при одновременных запросах
* **Описание:** При быстрой отправке нескольких параллельных запросов на покупку/активацию подписки бэкенд может создать несколько дублирующих записей в базе данных или списать баланс несколько раз.
* **Действия для воспроизведения (Шаги):**
  1. Подготовить скрипт, отправляющий 5 одновременных POST-запросов на `/api/v1/subscription/activate`.
  2. Запустить скрипт для аккаунта с балансом, достаточным только для одной подписки.
* **Фактический результат:** В БД создается несколько активных подписок, баланс пользователя уходит в отрицательное значение или списывается некорректно.
* **Ожидаемый результат:** Обрабатывается только первый запрос, остальные отклоняются с ошибкой (например, 409 Conflict) благодаря механизму транзакций БД или распределенной блокировке (Redis lock).

## 2. Блок: Лимиты и Конфиги (Limits & Configurations)

### BUG-003: Превышение лимита одновременных сессий (IP Limits) не блокирует доступ
* **Описание:** Механизм контроля лимита одновременных подключений не прерывает текущие сессии пользователя, если лимит был изменен администратором в меньшую сторону во время их активности.
* **Действия для воспроизведения (Шаги):**
  1. Установить в админке лимит одновременных подключений для подписки равным 3.
  2. Успешно подключить 3 устройства.
  3. В админке снизить лимит до 1 устройства.
* **Фактический результат:** Все 3 устройства продолжают работать без разрыва соединения до окончания срока действия подписки.
* **Ожидаемый результат:** Изменение лимита должно инициировать отправку команды на сервер раздачи (например, API Xray/Sing-box) для немедленного сброса лишних сессий (disconnect) и приведения их количества к новому лимиту.

---

### BUG-004: Сброс лимита трафика происходит по локальному времени сервера, а не по UTC
* **Описание:** Ежемесячный или ежедневный сброс использованного трафика (Limits reset) происходит по локальному времени часового пояса сервера, что вызывает рассинхронизацию, если пользователи находятся в других часовых поясах.
* **Действия для воспроизведения (Шаги):**
  1. Создать подписку с дневным лимитом 1 ГБ.
  2. Исчерпать лимит в 23:30 по UTC (на сервере в этот момент может быть 02:30 следующего дня).
  3. Дождаться 00:01 по UTC.
* **Фактический результат:** Лимит трафика сбрасывается только тогда, когда наступает полночь на самом сервере, а не в установленное единое время по UTC.
* **Ожидаемый результат:** Все расчеты лимитов времени и сбросов трафика должны выполняться строго в UTC-координатах на уровне кода и БД.

## 3. Блок: Панель администратора и Бэкенд (Admin Panel & Backend Integration)

### BUG-005: Повреждение структуры конфигурационного файла при сохранении некорректных символов в админке
* **Описание:** Ввод кириллицы или спецсимволов в конфигурационные поля админки (например, описание сервера или дополнительные параметры) ломает парсер конфигураций на стороне бэкенда или на стороне клиентов.
* **Действия для воспроизведения (Шаги):**
  1. Перейти в панель администратора -> Настройки конфигураций.
  2. В поле шаблона конфигурации ввести некорректную строку (например, незакрытые кавычки в JSON или YAML-формате).
  3. Нажать кнопку "Сохранить".
* **Фактический результат:** Настройки сохраняются в БД. При следующей генерации конфигов бэкенд падает с ошибкой парсинга (`JSONDecodeError` / `YamlParseException`), делая невозможным скачивание подписок для всех пользователей.
* **Ожидаемый результат:** Форма в админке должна валидировать вводимые данные (JSON/YAML схемы) перед сохранением в БД и возвращать ошибку валидации пользователю на фронтенде.

*К сожалению, на данный момент у меня нет возможности напрямую проанализировать исходный код вашего репозитория по ссылке `https://github.com/enver-isliamov/izi.net/tree/main` [8.1]. Это связано с тем, что автоматические инструменты сканирования могут блокироваться защитой GitHub (требующей авторизации) [8.1], а сам репозиторий не проиндексирован в поисковых системах.


# Проверка комплексной работоспособности системы

Для того чтобы после исправления всех багов гарантировать корректное взаимодействие всех слоев системы (сайт -> админка -> бэкенд -> выдача подписок -> обновление -> лимиты), необходимо верифицировать следующие интеграционные цепочки:

1. **Цепочка авторизации и инициализации:**
   * **Действие:** Регистрация пользователя на **сайте/фронтенде**, создание первой сессии, передача JWT-токена в бэкенд, отображение личного кабинета.
2. **Цепочка управления подписками:**
   * **Действие:** Заказ подписки на фронтенде -> Бэкенд фиксирует транзакцию, генерирует уникальный токен подписки -> Панель администратора мгновенно отображает новую активную подписку с базовыми лимитами.
3. **Цепочка доставки конфигураций:**
   * **Действие:** Клиент делает GET-запрос на обновление конфига по своей ссылке -> Бэкенд парсит шаблоны конфигов из базы, подставляет актуальные параметры серверов и лимиты пользователя, отдает корректный файл.
4. **Цепочка применения ограничений:**
   * **Действие:** Пользователь превышает лимит трафика -> Фоновый демон на бэкенде/ноде фиксирует перерасход -> Статус подписки меняется на `Exceeded` -> При попытке обновить конфиг выдается заглушка или ошибка, а текущее соединение разрывается.

# Рекомендации по тестам и автотестам

Поскольку проект состоит из фронтенда, бэкенда и системных служб раздачи конфигов, рекомендуется внедрить трехслойную пирамиду тестирования:

### 1. Модульные тесты (Unit Tests) для Бэкенда
* **Что тестировать:**
  * Валидацию генерируемых строк конфигураций (убедиться, что на выходе всегда валидный JSON/YAML/URI-строка).
  * Логику расчета лимитов (проверка математики вычисления остатка трафика и времени действия подписки).
* **Инструменты:** pytest (для Python) или JUnit/xUnit (в зависимости от используемого языка на бэкенде).

### 2. Интеграционные тесты (Integration Tests)
* **Что тестировать:**
  * Симуляцию конкурентных запросов (Race Condition) при покупке подписки.
  * Взаимодействие API с базой данных (правильность наложения транзакционных блокировок при изменении лимитов).
  * Корректность кэширования HTTP-ответов при выдаче обновлений подписок.

### 3. Сквозные автотесты (E2E Tests)
* **Что тестировать:**
  * Сценарий: Администратор в панели изменяет лимит пользователя -> Пользователь на фронтенде сразу видит новые лимиты -> API возвращает новый конфиг.
* **Инструменты:** **Playwright** или **Cypress**. Они позволяют имитировать реальные клики в админке и проверять изменения на клиентском сайте.

# Задания для команды разработки по решению проблем

Для систематического исправления багов и стабилизации проекта рекомендуется разбить работу на следующие задачи:

* [ ] **Задача 1: Внедрение строгой валидации конфигов в админке.**
  * *Решение:* Добавить JSON Schema / Pydantic / FluentValidation на бэкенд-эндпоинт сохранения настроек в админке.
* [ ] **Задача 2: Исправление работы с часовыми поясами.**
  * *Решение:* Провести аудит кода. Все операции сохранения дат (создание, продление, сброс лимитов подписок) перевести на `DateTime.UtcNow` / `timezone.now()` (UTC).
* [ ] **Задача 3: Предотвращение Race Condition при покупках.**
  * *Решение:* Обернуть логику списания баланса и выдачи подписки в транзакцию БД с уровнем изолированности `Serializable` или использовать распределенную блокировку на базе Redis.
* [ ] **Задача 4: Настройка кеширования заголовков эндпоинта подписок.**
  * *Решение:* Добавить middleware или декораторы, принудительно выставляющие заголовки `Cache-Control: no-cache, no-store, must-revalidate` для API-методов выдачи подписок.
