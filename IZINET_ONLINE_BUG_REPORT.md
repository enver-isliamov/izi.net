# Отчёт о тестировании izinet.online

**Дата тестирования:** 07.07.2026
**Тестировщик:** MiniMax Agent
**Версия сайта:** Неизвестна (SPA на Vite)
**Сайт:** https://izinet.online

---

## Резюме

В ходе тестирования пользовательского пути оформления подписки на VPN-сервис izinet.online было обнаружено **14 критических и высокоприоритетных багов**, препятствующих нормальному использованию сервиса. Наиболее серьёзной проблемой является невозможность различения страниц регистрации и авторизации, что создаёт критическое препятствие для новых пользователей. Также выявлены серьёзные уязвимости в системе безопасности, связанные с публичным раскрытием служебных ключей API.

## Критические баги (Critical)

### 1. Страницы /register и /login отображают идентичный контент

**Серьёзность:** CRITICAL
**Статус:** Воспроизводится
**Приоритет устранения:** НЕМЕДЛЕННО

**Описание проблемы:**
При переходе на страницы авторизации (`/login`) и регистрации (`/register`) пользователь видит абсолютно одинаковый контент. Обе страницы показывают заголовок "С возвращением" с описанием "Войдите в свой аккаунт для управления VPN", что создаёт критическую проблему для пользовательского опыта.

**Воспроизведение:**
Открытие страницы https://izinet.online/login в браузере демонстрирует форму входа с заголовком "С возвращением". Переход по ссылке на страницу регистрации (например, через навигацию или ввод URL https://izinet.online/register) отображает идентичный контент с тем же заголовком и описанием, вместо ожидаемой формы регистрации.

**Ожидаемое поведение:**
Страница `/register` должна отображать форму регистрации с полями для создания нового аккаунта (email, пароль, подтверждение пароля) и заголовком "Создайте аккаунт" или "Зарегистрируйтесь". Страница `/login` должна показывать форму входа с заголовком "Войдите в аккаунт".

**Фактическое поведение:**
Обе страницы показывают одну и ту же форму входа с описанием "Войдите в свой аккаунт для управления VPN".

**Влияние на бизнес:**
Критическое влияние на конверсию. Потенциальные клиенты не могут зарегистрироваться на сайте, что делает невозможным оформление подписки. Это приводит к полной потере пользователей, приходящих с маркетинговых каналов.

**Рекомендация:**
Необходимо проверить роутинг приложения (вероятно, React Router или аналогичная библиотека) и убедиться, что компоненты LoginPage и RegisterPage загружаются корректно. Также следует проверить серверную маршрутизацию на предмет корректной обработки SPA-роутов.

### 2. Страница /forgot-password отображает форму входа

**Серьёзность:** CRITICAL
**Статус:** Воспроизводится
**Приоритет устранения:** НЕМЕДЛЕННО

**Описание проблемы:**
При переходе на страницу восстановления пароля (`/forgot-password`) пользователь вместо формы сброса пароля видит стандартную форму входа в аккаунт. Функционал восстановления пароля полностью недоступен.

**Воспроизведение:**
Открытие URL https://izinet.online/forgot-password в браузере отображает идентичную страницу с формой входа вместо формы восстановления пароля с полем для ввода email.

**Ожидаемое поведение:**
Страница должна содержать форму восстановления пароля с полем для ввода email и кнопкой "Восстановить пароль" или "Сбросить пароль". Текст должен быть информативным: "Введите email, и мы отправим ссылку для сброса пароля".

**Фактическое поведение:**
Страница показывает форму входа с описанием "Войдите в свой аккаунт для управления VPN".

**Влияние на бизнес:**
Пользователи, забывшие пароль, не могут восстановить доступ к аккаунту. Это приводит к потере существующих клиентов и невозможности повторных покупок.

**Рекомендация:**
Реализовать корректную маршрутизацию для страницы восстановления пароля. В коде приложения присутствуют строки для функционала восстановления пароля ("Восстановление пароля", "Введите email, и мы отправим ссылку для сброса пароля", "Недействительная или устаревшая ссылка для восстановления пароля"), что указывает на то, что функционал был намеренно реализован, но не подключён к маршрутизации.

### 3. Публичное раскрыление service_role API ключа Supabase

**Серьёзность:** CRITICAL (Security)
**Статус:** Обнаружен
**Приоритет устранения:** НЕМЕДЛЕННО

**Описание проблемы:**
В исходном коде JavaScript приложения обнаружен service_role ключ Supabase, который предоставляет полный доступ к базе данных без ограничений. Данный ключ не должен присутствовать в клиентском коде.

**Расположение:**
Файл `/assets/index-K2C-HP63.js`, строка содержит JWT токен вида `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJzdXBhYmFzZSIsInJlZiI6InJ0eW51a2tvdWVxcHZlbWxzaGR4Iiwicm9sZSI6InNlcnZpY2Vfcm9sZSIs...`

**Ожидаемое поведение:**
В клиентском коде должен использоваться только `anon` ключ (публичный), который обеспечивает доступ только к разрешённым операциям через политики безопасности Row Level Security (RLS).

**Фактическое поведение:**
Service_role ключ, дающий полные права администратора, присутствует в клиентском JavaScript коде и доступен любому пользователю сайта.

**Влияние на безопасность:**
Критическая уязвимость. Злоумышленник может использовать service_role ключ для выполнения любых операций с базой данных: чтение, изменение и удаление данных всех пользователей, включая платёжную информацию, если таковая имеется.

**Рекомендация:**
1. Немедленно сгенерировать новый service_role ключ в панели управления Supabase.
2. Убедиться, что в клиентском коде используется только anon ключ.
3. Проверить, что все sensitive операции выполняются через серверные API, а не напрямую из браузера.
4. Рассмотреть возможность использования серверных функций Supabase для sensitive операций.

### 4. API endpoints возвращают ошибку 404

**Серьёзность:** CRITICAL
**Статус:** Воспроизводится
**Приоритет устранения:** НЕМЕДЛЕННО

**Описание проблемы:**
Прямые запросы к API endpoints аутентификации возвращают ошибку 404 Not Found, что делает невозможным программный вход и регистрацию пользователей.

**Протестированные endpoints:**
- `POST /api/auth/login` — 404
- `POST /api/auth/register` — 404
- `POST /auth/v1/` — 404
- `POST /auth/v1/login` — 404
- `GET /api/pricing` — 404

**Ожидаемое поведение:**
API endpoints должны возвращать корректные ответы (200, 400, 401) с JSON данными или соответствующими сообщениями об ошибках.

**Фактическое поведение:**
Все запросы возвращают HTML страницу ошибки с текстом "Cannot POST /api/auth/login" и кодом 404.

**Влияние на бизнес:**
Блокирована интеграция с внешними сервисами и автоматизация процессов. Также это может указывать на проблемы с маршрутизацией на стороне сервера.

**Рекомендация:**
1. Проверить конфигурацию сервера (Vercel, Nginx, Apache) на предмет корректной маршрутизации SPA приложения.
2. Убедиться, что все API routes корректно проксируются на backend.
3. Добавить обработку fallback для SPA роутов.

**Рекомендация:**

### 7. Недостаточные требования к паролю

**Серьёзность:** HIGH (Security)
**Статус:** Обнаружен в коде
**Приоритет устранения:** Высокий

**Описание проблемы:**
Система принимает пароли от 6 символов без дополнительных требований к сложности. Для VPN-сервиса, работающего с чувствительным трафиком пользователей, это неприемлемо.

**Расположение:**
Функция валидации пароля в `/assets/index-K2C-HP63.js`: `"Пароль должен содержать не менее 6 символов"`

**Ожидаемое поведение:**
Пароль должен соответствовать современным требованиям безопасности:
- Минимум 8 символов
- Минимум одна заглавная буква
- Минимум одна строчная буква
- Минимум одна цифра
- Минимум один специальный символ

**Фактическое поведение:**
Пароль принимается при длине от 6 символов без проверки сложности. Пользователь может использовать "123456" или "password" в качестве пароля.

**Влияние на безопасность:**
Слабые пароли легко подбираются методами brute-force или dictionary attack. Это создаёт риск компрометации аккаунтов пользователей и доступа к их VPN-трафику.

**Рекомендация:**
1. Усилить требования к паролю: минимум 8 символов, обязательно буквы разных регистров и цифры.
2. Добавить индикатор силы пароля в форме регистрации.
3. Реализовать защиту от brute-force (ограничение попыток входа, captcha после 3 неудачных попыток).

### 8. Некорректный формат ссылки на Telegram бота

**Серьёзность:** MEDIUM
**Статус:** Обнаружен в коде
**Приоритет устранения:** Средний

**Описание проблемы:**
В документации и на сайте используется формат `@izi_netbot` с нижним подчёркиванием, тогда как корректный формат Telegram — `@izi_netbot` с подчёркиванием, но в интерфейсе может требоваться ссылка `https://t.me/izi_netbot`.

**Расположение:**
Файл политики возвратов содержит: `Telegram-бот [@izi\_netbot](https://t.me/izi\_netbot)`

**Воспроизведение:**
Клик по ссылке `https://t.me/izi_netbot` работает корректно, однако сам текст ссылки `@izi_netbot` не является кликабельным в некоторых контекстах.

**Ожидаемое поведение:**
Все ссылки на Telegram ботов должны использовать полный URL формат `https://t.me/username` или корректный @mention.

**Фактическое поведение:**
Текст ссылки `@izi_netbot` отображается как есть, без автоматического преобразования в кликабельную ссылку.

**Рекомендация:**
Убедиться, что все ссылки на Telegram ботов используют полный URL формат `https://t.me/izi_netbot` для обеспечения корректной работы на всех платформах.

## Среднеприоритетные баги (Medium)

### 9. Отсутствие видимой валидации форм

**Серьёзность:** MEDIUM
**Статус:** Предполагается
**Приоритет устранения:** Средний

**Описание проблемы:**
В извлечённом контенте страниц авторизации не видно сообщений валидации для полей формы. Пользователь может ввести некорректные данные без получения немедленной обратной связи.

**Ожидаемое поведение:**
При вводе некорректного email должен появляться red border и сообщение "Введите корректный email". При пустом пароле — "Пароль обязателен". При несовпадении паролей — "Пароли не совпадают".

**Фактическое поведение:**
Сообщения валидации не отображаются в статическом извлечении контента.

**Рекомендация:**
Добавить inline валидацию с визуальными подсказками (red border, сообщения под полем) для всех полей форм.

**Серьёзность:** MEDIUM
**Статус:** Предполагается
**Приоритет устранения:** Средний

Кнопка остаётся активной, пользователь не получает обратной связи о процессе отправки.

**Влияние на UX:**

## Пользовательский путь (User Flow)

### Успешный путь (Happy Path)

1. **Главная страница** → Пользователь видит описание услуги
2. **Нажатие "Зарегистрироваться"** → Переход на страницу регистрации
3. **Заполнение формы** (email, пароль) → Создание аккаунта
4. **Email подтверждение** (если требуется) → Активация аккаунта
5. **Личный кабинет** → Просмотр тарифов
6. **Выбор тарифа** → Оформление подписки
7. **Оплата** → Получение доступа к VPN

### Заблокированный путь (Buggy Path)

1. **Главная страница** → Пользователь видит описание услуги
2. **Нажатие "Зарегистрироваться"** → Переход на `/register`
3. **КРИТИЧЕСКИЙ БАГ** → Пользователь видит форму входа вместо регистрации
4. **Попытка войти** → Неудача (учётных данных нет)
5. **Нажатие "Забыли пароль?"** → КРИТИЧЕСКИЙ БАГ: форма входа вместо восстановления
6. **Тупик** → Пользователь покидает сайт

## Приложение: Техническая информация

### Обнаруженные API Endpoints

**Публичные:**
- `GET /api/subscription/plans` — возвращает тарифы
- `GET /api/config` — возвращает конфигурацию (telegramBotName, publicUrl)

**Административные (обнаружены в исходном коде):**
- `/api/admin/diag`
- `/api/admin/payments`
- `/api/admin/payments/check-enot`
- `/api/admin/payments/confirm`
- `/api/admin/servers`
- `/api/admin/settings`
- `/api/admin/stats`
- `/api/admin/users`
- `/api/admin/users/create`

**Приватные:**
- `/auth/v1/` — Supabase Auth
- `/api/sub/` — Subscription
- `/api/subscription/buy`
- `/api/pay/create`
- `/api/promocode/apply`

**Тип подключения:** Wi-Fi / Mobile (Reality)

- **Frontend:** React (SPA на Vite)
- **Backend:** Supabase (Auth, Database)
- **Платёжная система:** ENOT.io
- **Мессенджер:** Telegram Bot (@izi_netbot)
